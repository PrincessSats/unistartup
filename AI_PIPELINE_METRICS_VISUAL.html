⚖️ Система LLM Судьи — Как это работает

🔄 Двухэтапный процесс оценки качества

Этап 1: Бинарные проверки (Validator)
Быстрая валидация структуры и функциональности. Если вариант не прошел — он отбрасывается и не идет дальше.

Этап 2: LLM-as-Judge (Reviewer)
Глубокая оценка качества по 5 измерениям. Запускается только для вариантов, прошедших все бинарные проверки.

✅ Этап 1: Бинарные Проверки (5 критериев)

Проверка Что проверяет Критерий прохождения Вес
FORMAT JSON валидность, наличие всех обязательных полей ✓ Все поля 1.0x
FUNCTIONAL Артефакт создан? Флаг валидный? Решение работает? ✓ Работает 1.5-2.0x
SOLVABILITY Решаема ли задача? Можно ли найти флаг с подсказками? ✓ Решаема 2.0x
NON_TRIVIALITY Задача интересна? Не слишком ли проста? ✓ Интересна 1.0-2.0x
RAG_GROUNDING Если используется RAG, задача соответствует контексту KB? ✓ Соответствует 1.5x
⚠️ Важно: Если вариант НЕ прошел хотя бы одну бинарную проверку, он немедленно отбрасывается и не подвергается LLM оценке.

🧠 Этап 2: LLM-as-Judge — 5 Измерений Качества

После прохождения бинарных проверок, вариант отправляется в LLM Reviewer (Yandex GPT), который оценивает качество по 5 независимым критериям. Каждый критерий оценивается от 0.0 до 1.0.

Измерение Что оценивает Примеры отличного (1.0) Вес
educational_value Учит ли задача реальным навыкам безопасности? Задача демонстрирует реальную уязвимость, учит защите 2.0x
scenario_realism Реалистичен ли сценарий? Встречается ли в реальности? Сценарий соответствует реальным инцидентам 2.0x
hint_quality Подсказки дозированы? Помогают ли найти путь? Подсказки направляют, не раскрывая ответ 2.0x
writeup_clarity Writeup понятен? Объясняет ли решение? Writeup ясный, с пояснениями каждого шага 2.0x
difficulty_calibration Соответствует ли сложность заявленному уровню? Easy = 5 мин, Medium = 20 мин, Hard = 60+ мин 2.0x

📊 Как рассчитывается итоговая оценка

Quality Score = (sum всех 5 измерений) / 5

Например: (0.9 + 0.8 + 0.85 + 0.9 + 0.8) / 5 = 0.85 (или 8.5/10)

Пороги:

  • > 0.75 (7.5/10) — очень хорошее качество
  • 0.65 - 0.75 — приемлемое качество
  • < 0.65 — нужно улучшение

🎯 GRPO: Сравнительный рейтинг вариантов

После оценки всех N вариантов в батче, система рассчитывает GRPO (Group Relative Policy Optimization) метрики:

Метрика Формула / Описание
Group Mean Reward Средняя награда всех вариантов = Σ reward_total / N
Group Std Reward Стд. отклонение = √(Σ(reward - mean)² / N)
Advantage Насколько вариант выше среднего = (reward - mean) / std
Rank in Group Позиция варианта в отсортированном списке (1 = лучший)
💡 Результат: Лучший вариант (Rank = 1) автоматически выбирается и отправляется на публикацию в Tasks.

🔄 Полный цикл: От генерации к публикации

1️⃣ Генерация
User запросил N вариантов → LLM генерирует N различных spec

2️⃣ Артефакты
Для каждого spec создается артефакт (файл, VM, HTML и т.п.)

3️⃣ Бинарные проверки
Validator проверяет FORMAT, FUNCTIONAL, SOLVABILITY, NON_TRIVIALITY
❌ Не прошел → выбросить, перейти к следующему

4️⃣ LLM Оценка
Reviewer оценивает только прошедших вариантов по 5 измерениям
Результат: quality_score (0.0-1.0)

5️⃣ GRPO Ранжирование
Сравнить все варианты, вычислить Advantage и Rank

6️⃣ Выбор лучшего
Variant с Rank = 1 → selected = True

7️⃣ Публикация
Лучший вариант → Task в базе, готов для пользователей ✅
🔧 SQL Примеры

Для аналитики качества:

SELECT task_type, difficulty, pass_rate, avg_quality FROM ai_generation_variants WHERE created_at > NOW() - INTERVAL '7 days' ORDER BY pass_rate DESC;

Для оптимизации стоимости:

SELECT temperature, COUNT(*), avg(tokens_input + tokens_output) AS avg_tokens FROM ai_generation_variants GROUP BY temperature ORDER BY avg_tokens;

Для отладки отказов:

SELECT failure_reason, COUNT(*) AS count FROM ai_generation_variants WHERE failure_reason IS NOT NULL GROUP BY failure_reason ORDER BY count DESC;

Запуск полного отчета:

python scripts/generate_metrics_report.py
